Cricket Workload and Injury Management System
I built this terminal based C++ application to track cricket player workloads and injury history. Coaches manage a roster of players. Players log their own training data. The goal is simple. Give teams a way to catch dangerous workload patterns before they turn into injuries.
User is the base class every account inherits from. It holds username, email, and role. It has no real behavior beyond getters, setters, and a logout stub. Its only job is to be the shared identity layer that Player and Coach build on top of. User has three constructors. A full constructor for account creation, a default constructor, and a username only constructor. That third one turns out to matter the most, and I explain why below.
Player extends User and represents an athlete. It tracks a name, age, and a PlayerType enum (FastBowler, Spinner, Batsman, WicketKeeper, FastAllRounder, SpinAllRounder). It holds a vector of every session logged, training, match, or recovery. It holds a vector of every injury reported, each one carrying a severity level, an expected recovery time, and a description. It also holds a single Plan object representing the player's current assigned training, made up of exercises, drills, and matches.
Coach extends User and represents the person managing a roster. It holds a coach ID, a name, and a vector of Player objects. Its methods act on a specific player pulled from that roster. It can view a player's details, view their sessions, view their plan, assign or overwrite a plan, add or remove a player directly, or pull an existing player account onto the roster just by their username.
Authentication, Kept Separate on Purpose
The most deliberate decision in this project is keeping authentication and object construction as two completely separate jobs.
auth.cpp does one thing. Given a username and password, it reads credentials.txt line by line, splits each line into a stored username, password hash, and role, then checks the password using bcrypt validation. If it matches, it hands back a role. It has no idea what a Player or Coach even is.
Once main.cpp gets that role back, it decides which class to build. This is where the username only constructor earns its place. Calling Player with just a username does not create an empty object waiting to be filled in. Its constructor immediately loads the matching JSON file and pulls every field back into memory, sessions, injuries, training plan, all of it. Coach works the same way with its own JSON file.
Logging in and loading a user's data are two different questions. Is this really you. What does your account actually contain. Two different pieces of code answer those questions, and they never need to know about each other. That separation means I could swap in a real login system later without touching how player or coach data loads.
Passwords Are Never Stored As Plain Text
Account creation and login both run through a real bcrypt implementation, not a placeholder. I vendored the actual OpenBSD bcrypt and Blowfish cipher source directly into the project. This is the same hashing approach used in real production systems, and it means the project has zero dependency on OpenSSL or any external crypto library. credentials.txt never holds a password. Only a username, a bcrypt hash, and a role.
Data Persistence Without a Database
Every Player and Coach account saves to its own JSON file using nlohmann json. The username doubles as the filename and the lookup key, the same pattern real document databases use, just without a server behind it. Every action that changes data, logging a session, reporting an injury, updating a plan, adding or removing a player, saves immediately after it runs. Memory and disk never fall out of sync during a session.
The Terminal Interface
main.cpp handles account creation with duplicate username protection, login, and two separate dashboard loops.
The player dashboard has six options. View sessions. Log a new session with type, duration, and notes. View the current plan. Report an injury with a severity level and recovery estimate. Review past injuries. View your own profile.
The coach dashboard also has six options. View a player's details. View a player's sessions. View a player's plan. Assign or update a plan, built step by step by entering exercises, drills, and matches one at a time. Remove a player from the roster. Pull an existing player onto the roster using their username.
Both dashboards run on a loop with input validation on every prompt. Bad input clears the failed state and asks again instead of crashing.
How To Build It
git clone https://github.com/BhamSaath/cricket-workload-tracker
 cd cricket-workload-tracker
 g++ std=c++14 main.cpp auth.cpp Player.cpp Coach.cpp User.cpp bcrypt.cpp blowfish.cpp o cricket
 ./cricket
Nothing needs to be installed separately. The json library and the bcrypt source are already sitting in the repo, so this builds the same way on Mac, Linux, or WSL as long as you have a C++14 compiler.
What This Project Actually Taught Me
I started this knowing basic C++. Loops, conditionals, simple classes. By the end I had built and debugged a multi file inheritance hierarchy, a JSON persistence layer that saves and reloads nested objects correctly, and a real cryptographic library wired in from source. I also learned a few things the hard way. Mixing cin and getline without clearing the input buffer will quietly corrupt your next input. A constructor should never silently touch the file system without you knowing exactly why. And the gap between code that compiles and code that survives an actual person typing into it is bigger than it looks until you test it yourself.
